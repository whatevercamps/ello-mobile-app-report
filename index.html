vista 1.1: autenticación

Aquí, si es la primera vez que el usuario se registra con su número telefónico, esta vista se despliega solicitando al usuario su información básica de perfil. Así, el usuario puede detallar su nombre, su imagen de perfil, y una breve descripción del mismo.

En esta vista el usuario puede editar la configuración básica de funcionamiento de la aplicación, así como su información básica de perfil.

En esta vista el usuario puede conversar con otros usuarios. Puede adjuntar elementos de texto, audio, imagen, o video.

Threads Y cómo éstos afectan el performance

AvatarOpener (fallo crítico) Contactos Chatting

La librería AvatarOpener maneja threads en dos partes importantes: cuando ya se tiene la imagen después de editarla y se quiere guardar, y cuando se tiene la imagen inicial pero se quiere borrar. Aunque para la última sólo se usa para mostrar el ActivityIndicator. Todo esto está en el delegado de AvatarOpener

Para los dos casos se muestra un ActivityIndicator en el centro de la imagen, para esto se hizo una extensión de UIImageView ubicada en Support/Public en la clase Public. La cual primero hace los ajustes de tamaño y constrains del activityIndicator y luego lo muestra mediante un DispatcQueue.main.async. Esto se hace de esta manera (async) debido a que la acción no requiere esperar a que el anterior bloque termine sino que se ejecuta directamente.

Para el caso de actualizar la foto del usuario se llama al método updateUserProfile. Este método lo que hace principalmente es crear una imagen miniatura llamando al método creteImageThumbnail de la clase public. Luego crea un DispatchGroup y manda dos threads, un thread persiste la imagen grande y el otro la imagen en miniatura.

A su vez, el uso actual de Threads en la librería está generando un fallo en el flujo normal de la aplicación, haciendo que ésta quede en espera prologanda o permanente. Esto ocasiona que el usuario no pueda usar la aplicación indefinida. La buena noticia es que logramos identificar el error y hacer los cambios para que esta falla no ocurra más. Si desea saber al respecto, siga el siguiente enlace 🤓👉🏻 arreglo del fallo


Se usan threads para obtener todos los contactos del teléfono.

Se usan threads para verificar cuáles de los contactos cuentan con un teléfono registrado en la base de datos. Esta función puede ser asíncrona o sincrónicamente.

Es importante resaltar que así se tengan implementadas formas asíncronas y sincrónicas para sincronizar los contactos con los registrados, siempre se utiliza la implementación asincrónica. Como se evidencia en la imagen de abajo la función fetchFalconUsers es la que contiene los dos métodos. Esta función como se puede ver solo se llama en dos ocaciones y las dos usan como parámetro: “asynchronously: true”

Se usa la clase ViewPlaceHolder para mostrar mensajes en caso de que haya algún problema como que no se puede acceder a los contactos o para informar al usuario cosas como que no ha iniciado ningún chat. Esta consiste en un título y un subtítulo que se muestra. Se usan threads para quitar o poner el placeholder en las vistas.


Se usan threads cuando se está creando un nuevo grupo para ocultar el indicador de carga, enviar el mensaje notificando a todos los usuarios del nuevo grupo y volver a la pantalla principal de chats.

Cuando se envían elementos multimedia se utilizan threads para cargar cada uno a la base de datos.

Antes de comenzar

Falcon Messenger es una aplicación diseñada para ser la perfecta alternativa de mensajería instantánea de código abierto para iOS. Esto permite que tenga funcionalidades muy interesantes implementadas, gracias a los diversos aportes que la comunidad le ha brindado.

La funcionalidad principal de Falcon Messenger es comunicar personas a través de la aplicación únicamente con un número telefónico y acceso a internet.

¿Cómo funciona la autenticación?

1. Se introduce el número y se selecciona la ciudad. Estos datos se pasan a la siguiente vista en un label.

2. Cuando se presiona Next se comprueba si hay conexión a internet y si no se ha enviado ningún código de verificación antes y se procede a enviar el mensaje de verificación.

3. Para el mensaje se llama al método PhoneAuthProvider .provider() .verifyPhoneNumber el cual envía el mensaje de texto al número indicado y muestra un SFSafariViewController para comprobar que es una persona y no un ataque. Al final guarda en userDefaults una variable verificationID que es con la que posteriormente se va a comprobar si el código que ingresó el usuario es correcto.

3. Si el usuario presiona el botón de resend message se vuelve a repetir lo mismo del paso anterior a excepción que el método ahora deshabilita el botón y se inicia un contador al final.

4. Posteriormente cuando ya el usuario introduce un código de confirmación y presiona next, se accede a la función authenticate la cual accede a la VerificationID guardada y junto con el código que ingresó el usuario crea una credencial para acceder y lo intenta con Auth.auth().signIn de Firebase. Si termina en error lo notifica al usuario. Si es exitoso continúa verificando si el usuario ya se ha registrado y pasa a la siguiente pantalla.

5. En la última pantalla el usuario pasa tiene que ingresar el nombre, un estado y una foto. Si el usuario ya existe en la base de datos, carga los datos que ya se tenían con el método checkIfUserDataExist en el que consulta en Firebase y asigna los valores. Para la foto la escala primero y luego la muestra.

¿Cómo funciona la vista de configuración?

La clase AccountSettingController es la clase principal, contiene un TableView para mostrar todas las opciones. Se muestra el contenido dividido en dos secciones: La primera sección contiene todas las opciones posibles a editar por el usuario y la segunda sección solo contiene el botón de cerrar sesión.

Es importante resaltar que se tiene un observador que está pendiente si hay algún cambio remoto en la base de datos y actualiza la casilla correspondiente en tiempo real.

Se usa la librería AvatarOpener para acceder a las fotos o tomar una foto nueva y posteriormente editarla. Para esto está la clase AccountSettingsAvatarOpenerDelegate

Se guarda en UserDefaults la selección del usuario si desea tener que autenticarse con Código, TouchID o FaceID según sea el caso.

Para borrar los documentos y todos temporales se usa el método FileManager.default.clearTemp()

Para borrar el caché se llama a la librería SDWebImage la cual tiene el método clearDisk() y clearMemory() que ayudan a eliminar todo el caché.

¿Cómo funciona la vista de lista de chats?

Para mostrar la lista de chats se usa un table view configurado en la clase ChatsTableViewController - Se tiene un observador que está escuchando el tema actual que tiene el usuario.

Al iniciar se usa el método fetchConversations() de la clase ConversationsFetcher para recuperar todas la conversaciones del usuario - Para mantener las conversaciones actualizadas se usa el protocolo ConversationUpdatesDelegate

¿Cómo funciona la vista de chating?

Para cargar los mensajes anteriores del chat, se usa la clase ChatLogHistoryFetcher que tiene el método loadPreciousMessages.

Para mostrar la grabación de voz se utiliza una onda mediante la clase SwiftSiriWaveFormView que se encuentra en Support/WaveformView. Ésta es una adaptación del repositorio: github.com/stefanceriu/SCSiriWaveformView . En general es una implementación de una onda parecida a la que aparece cuando se usa Siri, la cual cambia la amplitud según el nivel de la voz.

Para el envío de mensajes se tiene la clase MessageSender, la cual principalmente tiene métodos de: sendTextMessage(), sendPhotoMessage() y sendVideoMessage() para el envío de los diferentes tipos de mensajes.

Se selecciona el tipo de celda según si el archivo media se seleccionó es una imagen o no.



¿Cómo funciona la sincronización de contactos?

La vista principal se da mediante la clase ContactsController la cual en primer lugar llama a la clase ContactsFetcher la cual accede a los contactos del usuario usando threads asíncronos y crea una variable con los contactos.

También se usa la clase FalconUserFetcher en concurrencia asíncrona para leer todos los contactos en el teléfono, leer los números que se tiene en firebase y comprobar que contactos.

La vista se muestra en un TableView con 3 secciones, en la primera sección se tiene un chat personal, en la segunda se tiene todos los contactos del usuario que también son usuarios de Falcon y en la tercera el resto de contactos.

Si se selecciona un contacto que no use Falcon se llama a la clase ContactsDetailController donde se muestra una vista con un TableView con el nombre, número de teléfono y la opción de enviar un mensaje

Para el manejo del buscador se accede a la clase ContactsController + SearchHandlers para manejar la búsqueda en tiempo real filtrando los usuarios que se muestran en cada sección del table view.



Si el contacto si está registrado en falcon se accederá a la parte de chat con el contacto que se explicará posteriormente.
Para el manejo de las celdas, se sobrescribe el método prepareForReuse para asegurar que todas las celdas reutilizadas tengan los colores que se requiere para el tema actual.

Cuando se van a sincronizar los contactos se comprueba primero si hay permiso del usuario para acceder a los contactos del teléfono.



Cuando no se tiene permitido el acceso a los contactos se muestra un mensaje mostrando el problema que se tiene y cómo solucionarlo.

¿Cómo funciona la sincronización de contactos?

Para la conectividad eventual se utiliza la clase Public en la carpeta Support/Public

El estado de la conexión se almacena en una variable que puede tomar 3 estados: notReachable cuando no se puede establecer la conexión, reachableViaWan cuando se establece conexión mediante los datos del celular y reachableViaWifi cuando se puede establecer conexión mediante Wifi.

Para revisar el estado de la conexión se usa SCNetworkReachabilityFlags valores si se puede alcanzar o no una dirección dada.

Cuando el número de teléfono es demasiado largo o demasiado corto y se continúa con el proceso de autenticación la aplicación muestra una alerta pero no es clara y después de cerrarla no se puede editar el número de teléfono.

Error fatal al cargar imágenes
El fallo se presenta cuando el usuario intenta editar su foto de perfil, ya sea por cámara o desde su galería de imágenes. Esto se da por un error al manejar los distintos Threads involucrados en la tarea. Aquí se muestra la vista bloqueada que percibe el usuario en esta situación

Adcionalmente, investigamos las posibles causas y la solución del fallo. Esta información la puede encontrar en la sección de Threads siguiente este enlace.
Error fatal al cargar imágenes
Los mensajes se guardan en Firebase sin ningún tipo de encriptación, esto es un problema grave de seguridad y privacidad ya que si alguien accede a la base de datos puede ver las conversaciones de todos los usuarios.

Además, resulta interesante que una de las funcionalidades principales de Firebase es el almacenamiento temporal sin conexión; sin embargo, no hacen uso de éste para visualizar el envío de mensajes sin conexión. Cuando no se tiene conexión no es posible enviar los mensajes. Es decir, la aplicación impide la función lo cual debería ser manejado de m ejor manera.
